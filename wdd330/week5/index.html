  • Notes

    • Chapter 10: Testing and Debugging

    • Errors
      • System error - system or external device error
      • Programmer error - program contains incorrect syntax or faulty logic
      • User error - incorrect data, program couldn't handle
    • Exceptions - errors that return a value that the program can deal with
    • Stack Traces - the sequence of functions that lead to the error (listed backwards)
    • Warnings - possibly an error even though the program continues to run
    • To use strict mode, simply add 'use strict'; to the beginning of the script, or to use it on a per function basis, add the line as the first inside a function
    • Best practice is to use strict mode inside of a self-invoking function
      (function() {
      'use strict';
      // All your code would go inside this function
      }());
    • JavaScript modules use strict mode by default (no declaration is required)
    • Linting tools - test quality and force programming style JS Lint JS Hint ES Lint
    • Feature detection is the recommended way of determining browser support
    • Test for a feature inside of an if block:
      if (window.holoDeck) {
      virtualReality.activate();
      }
    • Modernizer can also be used to implement feature detection as well as Can I Use? which is useful for checking browser support
    • Debugging - breakpoints, alert(), console.log(), console.trace()
    • Using the debugger; keyword in a program will halt execution
    • Error Objects
      • The host environment can create an error object OR one can be created with: const error = new Error(...); the constructor takes a string to use as the error message
      • EvalError - only used for backwards compatibility to identify erros when using global eval()
      • RangeError - thrown when a a number is outside an allowable range
      • ReferenceError - thrown when a refernce is made to an item that does not exist
      • SyntaxError - thrown when the code uses incorrect syntax
      • TypeError - thrown when there is an error in the type of value used
      • URIError - thrown when there is a problem encoding or decoding a URL
      • InternalError - thrown when there is problem with the JavaScript engine, non-standard, common cause is too much recursion
    • Each error object above can be created with a customer constructor message
    • Safe to use (common) properties include: name, message, stack
    • Handle errors gracefully by catching them: try, catch, finally
    • Code inside a catch block will only run if an error occurs
      try {
      ...
      }
      catch(error) {
      ...
      }
    • finally {} can be added after a catch block if you want to run code regardless of whether the try succeeds or fails
    • Test-driven Development (TDD) TDD is dead. Long live testing.
      • Write test that initially fail
      • Write code to pass the tests
      • Refactor
      • Test refactored code
      • Write more tests for new features
    • Testing frameworks like Jest can make testing easier
